# ДЗ 4: Notification platform для онлайн банка

## Функциональные требования
1. Банковские сервисы отправляют уведомления через единый API (Notification platform), без прямого обращения к провайдеру
2. Система должна поддерживать транзакционные уведомления (зачисления и списания через push/SMS/email)
3. Рассылки должны поддерживать массовую отправку
4. Пользователь должен иметь возможность управлять уведомлениями: можно отключить рассылки и сервисные уведомления, транзакционные уведомления остаются обязательными
5. Система хранит историю доставки уведомлений (статус: отправлен/доставлен/прочитано/ошибка) для аудита и аналитики
6. Notification Platform поддерживает персонализацию шаблонов уведомлений (имя, сумма, баланс)
7. Система предотвращает дублирование одинаковых уведомлений одному пользователю в течение 5 минут

## Нефункциональные требования
1. Транзакционные уведомления доставляются с задержкой ≤ 3 секунд
2. Система обрабатывает пиковую нагрузку 3000 TPS (транзакционные уведомления)
3. Критичные уведомления доставляются хотя бы по 1 каналу в 99.9% случаев
4. Затраты на SMS не превышают 20% от общего бюджета уведомлений
5. Failover на резервный канал происходит ≤ 10 сек после отказа
6. Доступность платформы 99.95% (≤ 4.38 часа простоя/месяц)

## Архитектурно значимые требования (ASR)

### Низкая задержка критичных уведомлений
**Связные требования**:
    * Поддержка транзакционных уведомлений
    * Задержка транзакционных уведомлений ≤ 3 сек

**Почему влияет на архитектуру**: для минимизации времени задержки уведомлений требуются минимизация количества прыжков между сервисами => важные уведомления обрабатываются **в первую очередь** через отдельную быструю очередь, настройки пользователя хранятся **в оперативной памяти** (чтобы не обращаться к основной базе), критичные уведомления идут **по отдельному пути** без лишних проверок

**Архитектурные последствия**:
    * Брокер сообщений с приоритетными топиками (например, Kafka)
    * Быстрый кэш с настройками пользователя (Redis)
    * Отдельная группа обработчиков только для критических уведомлений
    
### Гарантированная доставка с failover
**Связные требовнаия**:
  * Поддержка транзакционных уведомлений
  * Критичные уведомления доставляются хотя бы по 1 каналу в 99.9% случае
  * Failover на резервный канал происходит ≤ 10 сек после отказа
**Почему влияет на архитектуру**: Требуется машина состояний для отслеживания всех попыток => каждая отправка записывается в базу атомарно и у каждой отправки уникальный идентификатор для предотвращения дублей

**Архитектурные последствия**:
  * Машина состояний и хранение этих состояний
  * Автоматическое отключение неработающих провайдеров
  * Проферка уникальности сообщения перед отправкой
  * Повторная попытка отправки с увеличивающимися интервалами

### Масштабируемость под разные типы нагрузки
**Связанные требования:** 
  * Рассылки должны поддерживать массовую отправку
  * Система обрабатывает пиковую нагрузку 3000 TPS (транзакционные уведомления)
  
**Почему влияет на архитектуру:**  
**Разные типы нагрузки** (резкие всплески транзакций и большие рассылки) => требуются отдельные каналы обработки и горизонтальное масштабирование

**Архитектурные последствия:**
  * Разные очереди: для быстрых важных и медленных массовых
  * Возможность добавлять новые серверы обработки под нагрузку
  * Пакетная отправка для массовых рассылок
  * Ограничение уведомлений на пользователя

## Ключевые архитектурные вопросы (AQ)

### Как обеспечить доставку критичного уведомления хотя бы по одному каналу?
**Порождающие требования**:
    * Гарантированная доставка с failover
    * Критичные уведомления доставляются хотя бы по 1 каналу в 99.9% случае
    * Failover на резервный канал происходит ≤ 10 сек после отказа

**Почему важно**:
Без ответа на этот вопрос невозможно гарантировать доставку уведовления => могут быть сбои отправки транзакционных и критических уведомлений => рост жалоб клиентов и оттток пользователей

### Как обеспечить быструю доставку при массовых рассылках?
**Порождающие требования**:
  * Низкая задержка критичных уведомлений
  * Гарантированная доставка с failover
  * Система обрабатывает пиковую нагрузку 3000 TPS (транзакционные уведомления)

**Почему важно**:
Массовые рассылки не должны замедлять критические уведомления => нужен баланс распределения нагрузки

### Как хранить настройки пользователя для мгновенного доступа при отправке?
**Порождающие требования:**
  * Низкая задержка критичных уведомлений 
  * Пользователь управляет уведомлениями 
  * Транзакционные уведомления ≤ 3 сек

**Почему важно:**
Каждое обращение к основной базе данных при отправке = +500мс задержки. При 3000 TPS это критично для соблюдения SLA.

## Архитектурные последствия ASR

### Низкая задержка критичных уведомлений
- Брокер сообщений с отдельными очередями для важных уведомлений
- Кэш настроек пользователей в оперативной памяти (Redis)
- Отдельная группа серверов только для обработки критичных уведомлений
- Минимальное количество сервисов в цепочке доставки

### Гарантированная доставка с failover
- База данных для хранения статусов всех уведомлений
- Система автоматического отключения неработающих провайдеров
- Уникальные идентификаторы для исключения дублей
- Логика повторных попыток с увеличивающимися паузами

### Масштабируемость под разные типы нагрузки
- Разные очереди обработки: быстрые для критичных, медленные для массовых
- Возможность добавлять новые серверы под нагрузку
- Пакетная отправка для больших рассылок
- Ограничение количества уведомлений на одного пользователя

## Архитектурные решения, которые НЕ подходят

### Синхронная отправка уведомлений из бизнес-сервисов
**Нарушает ASR:** Низкая задержка критичных уведомлений  
**Почему не подходит:** Сервис переводов ждет 3+ сек ответа от провайдера push => вся операция перевода тормозит => пользователи жалуются на несвоевременные уведомления об операциях

### Единая очередь для всех типов уведомлений
**Нарушает ASR:** Масштабируемость под разные типы нагрузки 
**Почему не подходит:** Маркетинговая рассылка на 1 миллион пользователей блокирует очередь => критичные уведомления о переводах задерживаются сверх 3 секунд

### Хранение настроек только в основной базе данных
**Нарушает ASR:** Низкая задержка критичных уведомлений
**Почему не подходит:** Чтение настроек из Postgres при каждой отправке = +500мс => 3000 TPS × 500мс = перегрузка БД => нарушение SLA в 3 секунды

## Неопределённости и архитектурные риски

### Какие каналы доступны конкретному пользователю?
**Что неизвестно:** Установлено ли мобильное приложение (push), подтвержден ли номер телефона (SMS), актуален ли email.  
**Как проверить:** Анализ текущей user base — % пользователей с push-токенами, % с верифицированными номерами/почтой.

### Точное время отклика внешних провайдеров  
**Что неизвестно:** Push провайдеры дают ответ за 0.5-5 сек, SMS — 1-30 сек, email — до 60 сек. Точные SLA провайдеров.  
**Как проверить:** Load testing реальных провайдеров (Firebase, SMS.ru, SendGrid) + измерение p95/p99 latency.

### Какие уведомления считать успешными?  
**Что неизвестно:** Push "отправлен" = доставлен? SMS "доставлен" = прочитан? Нужны ли read receipts?  
**Как проверить:** A/B тест с реальными пользователями + анализ жалоб по типам уведомлений
