## Контекст

Текущая ситуация: Каждая команда банка (платежи, кредиты, карты) отправляет транзакционные уведомления самостоятельно через разные провайдеры. При сбоях push-уведомлений (Firebase downtime) пользователи не получают подтверждение переводов → рост жалоб, снижение retention.

Проблема: Нет механизма гарантированной доставки "хотя бы по одному каналу" с автоматическим переключением push→SMS→email.

Цель RFC: Спроектировать подсистему в Notification Platform, которая гарантирует доставку критичных уведомлений любой ценой, минимизируя стоимость (SMS в 10 раз дороже push).

### Ключевые вопросы
- Проблема: Как обеспечить доставку критичных уведомлений при сбоях провайдеров?
- Почему сейчас: Рост жалоб на 30% из-за недоставленных уведомлений о переводах
- Затронуты: Все 3 миллионы DAU + бизнес-сервисы банка

## Пользовательские сценарии

| Приоритет | Тип сценария | Действующее лицо | Сценарий |
|-----------|--------------|------------------|----------|
| MUST HAVE | Happy path | Сервис переводов | Сервис переводов отправляет уведомление => push доставлен за ≤ 3 сек |
| MUST HAVE | Failover | Сервис переводов | Push провайдер недоступен => SMS отправлен за 8 сек => статус DELIVERED |
| SHOULD HAVE	| User settings |	Пользователь |	Пользователь отключил SMS => система отправляет только push + email |
| COULD HAVE | Read receipt |	Пользователь | Push прочитан => статус обновлен в реальном времени |

**Приоритеты:**
- **MUST HAVE** — обязательно к реализации
- **SHOULD HAVE** — желательно реализовать
- **COULD HAVE** — опционально, при наличии ресурсов

## Требования

### Функциональные требования

| № | Приоритет | Обозначение | Требование |
|---|-----------|-------------|------------|
| 1 | **MUST HAVE** | **FR1** | Гарантированная доставка транзакционных уведомлений хотя бы по 1 каналу (push/SMS/email) |
| 2 | **MUST HAVE** | **FR2** | Автоматическое переключение на резервный канал при ошибке/таймауте основного |
| 3 | **MUST HAVE** | **FR3** | Хранение полного аудита попыток доставки (канал, статус, время, результат) |
| 4 | **SHOULD HAVE** | **FR4** | Учет пользовательских настроек при выборе каналов (отключены ли SMS/push) |
| 5 | **SHOULD HAVE** | **FR5** | Предотвращение дублей при failover (одно событие = одно уведомление) |

### Нефункциональные требования

| № | Приоритет | Обозначение | Требование |
|---|-----------|-------------|------------|
| 1 | MUST HAVE | **NFR1** | Первая попытка доставки ≤ **3 сек** (p99) |
| 2 | MUST HAVE | **NFR2** | Failover на резервный канал ≤ **10 сек** |
| 3 | MUST HAVE | **NFR3** | Успешная доставка хотя бы по 1 каналу: **99.9%** |
| 4 | MUST HAVE | **NFR4** | Максимум **0.1%** дублирующихся уведомлений |
| 5 | SHOULD HAVE | **NFR5** | SMS отправляются ≤ **20%** от общего объема |

**Расчёт нагрузок:**
- DAU: 3M × 2 транзакционных/день = 6M уведомлений/день = 70 TPS средняя
- Пик: 300K concurrent × 2 ÷ 3600сек = 2000 TPS
- Массовые кампании: до 1M пользователей = 15M сообщений за раз

## Варианты решения

### Вариант 1: Centralized Orchestrator (State Machine)

Описание: Единый оркестратор управляет состоянием уведомления (PENDING -> SENT -> DELIVERED/FAILED), автоматически переключает каналы по приоритету push -> SMS -> email

#### Архитектура

##### Описание

Notification Platform построена как распределённая система с асинхронной обработкой через брокер сообщений

Notification API

Принимает запросы от банковских сервисов и публикует события в очередь. Не выполняет доставку синхронно

**Orchestrator**

Управляет жизненным циклом уведомления:
выбирает канал (push → SMS → email), учитывает настройки пользователя и выполняет failover при ошибках

NEW → PUSH_SENT → (SUCCESS | FAIL)
FAIL → SMS_SENT → (SUCCESS | FAIL)
FAIL → EMAIL_SENT → (SUCCESS | FAILED_FINAL)

**Kafka**

Обеспечивает асинхронную обработку и буферизацию нагрузки

Разделяет потоки (критичные и массовые уведомления)

**Delivery Workers**

Отправляют уведомления через конкретные каналы (push, SMS, email) и возвращают результат

**Provider Adapters**

Инкапсулируют работу с внешними провайдерами и нормализуют ответы

**Redis**

Хранит настройки пользователей для быстрого доступа (без обращения к БД)

**PostgreSQL**

Хранит статусы уведомлений, историю отправок и используется для дедупликации

#### Этапы реализации

| Этап | Описание                                                                                    | Планируемый срок | Ресурсы          | Риски                                      |
| ---- | ------------------------------------------------------------------------------------------- | ---------------- | ---------------- | ------------------------------------------ |
| 1    | **MVP: базовая доставка (push)** — (API, Kafka, Worker без оркестратора)                    | 2 недели         | 2 backend        | Нет гарантий доставки, нет failover        |
| 2    | **Внедрение Orchestrator + state machine** — хранение статусов, управление жизненным циклом | 2–3 недели       | 2 backend        | Ошибки в логике состояний => дубли/потери  |
| 3    | **Подключение SMS и email каналов**                                                         | 2 недели         | 2 backend        | Нестабильность внешних провайдеров         |
| 4    | **Реализация failover** — таймауты, retries, переключение каналов                           | 2–3 недели       | 2 backend        | Неправильные тайминги => дубли или задержки |
| 5    | **Кэш пользовательских настроек (Redis)**                                                   | 1 неделя         | 1 backend        | Неконсистентность с БД                     |
| 6    | **Дедупликация и идемпотентность**                                                          | 1–2 недели       | 1 backend        | Ложно-положительные дубли                  |
| 7    | **Разделение очередей и масштабирование**                                                   | 1–2 недели       | backend + DevOps | Неправильная настройка Kafka               |
| 8    | **Мониторинг и алертинг**                                                                   | 1 неделя         | DevOps           | Недостаточная наблюдаемость                |
| 9    | **Нагрузочное тестирование**                                                                | 1–2 недели       | QA + backend     | Не выдержит 3000 TPS                       |


#### Преимущества
- Централизованный контроль доставки и failover
- Простая реализация гарантированной доставки (через state machine)
- Хорошая наблюдаемость (все статусы в одном месте)
- Легче обеспечить дедупликацию

#### Недостатки
- Оркестратор — потенциальная точка отказа
- Более высокая сложность реализации (вся система)
- Ограниченная гибкость при добавлении новых сценариев
- Может стать bottleneck при росте нагрузки

### Вариант 2: Event-Driven Workers (Separate Pipelines)

> **Описание:** [Краткое описание подхода]

#### Архитектура
Notification API — точка входа, публикует все критичные уведомления в общую очередь critical.notifications

Push Worker, SMS Worker, Email Worker — независимые сервисы, каждый:

Подписан на общую очередь Kafka

Проверяет Redis lock перед обработкой

Выполняет отправку только если lock свободен

Помечает уведомление как DELIVERED в Postgres

Redis Distributed Lock — гарантирует, что только один worker обрабатывает уведомление

#### Преимущества
- Хорошо масштабируется (каждый сервис отдельно)
- Нет единой точки отказа
- Легко добавлять новые каналы
- Асинхронная обработка выдерживает пики нагрузки

#### Недостатки
- Сложно гарантировать доставку и failover
- Трудно отлаживать цепочки событий
- Возможны дубли из-за асинхронности
- Нет единого состояния уведомления

## Выводы

**Рекомендация:** Centralized Orchestrator

**Обоснование выбора:**
В контексте банковской системы ключевыми являются:

- гарантированная доставка уведомлений
- строгий контроль failover
- аудит всех попыток доставки
- выполнение SLA по задержке и надёжности

Вариант с оркестратором лучше соответствует этим требованиям, так как обеспечивает централизованное управление состоянием уведомлений, упрощает реализацию failover и дедупликации и позволяет гарантировать SLA 99.9% доставки

Event-driven подход даёт лучшую масштабируемость, но увеличивает сложность обеспечения консистентности и предсказуемости поведения системы, что критично для транзакционных уведомлений.
